    • Categories of Logic in a Scalable React App

    • My Blog

    Overview

    In large, real-world applications (like those at Google, Facebook, or enterprise teams), developers separate much more than just UI logic and business logic. Here's a full breakdown of the different types of logic and code concerns that experienced engineers typically isolate.

    Categories of Logic in a Scalable React App

    Logic Type Description
    UI Logic Rendering, component structure, styles, layout
    Business Logic App rules, form workflows, permissions, what should happen
    Data Logic API requests, caching, normalization, RTK Query or React Query
    State Management Logic Global or shared state (e.g., Redux, Zustand, context)
    Auth/Access Logic What users can see/do, roles, feature flags
    Validation Logic Form field validation, custom rules, conditional validation
    Utility Logic Date formatting, string manipulation, calculations
    Side-Effect Logic Effects like useEffect, subscriptions, timers, scroll events
    Integration Logic Working with SDKs, analytics, 3rd party services (Stripe, Firebase)
    Workflow/State Machine Logic Multi-step forms, wizards, conditional flows
    Content/Copy Logic Dynamic labels, localization, pluralization (i18n)

    Real Example Breakdown (from a Facebook-style "Post Creator")

    Imagine a huge component like this:

    <PostForm />

    It might contain all this:

    Logic Concern Example Where It Belongs
    UI Logic <TextArea />, buttons, error display PostForm.jsx
    Business Logic "User must be verified" check usePostForm.js
    Data Logic createPost() mutation postApi.js (RTK Query or fetch)
    Validation Logic Check character count, tag limit formValidation.js
    Auth Logic Hide certain fields for banned users authContext.js or authUtils.js
    Utility Logic Format hashtags, strip HTML utils/formatPost.js
    Side Effects useEffect to fetch drafts on load usePostForm.js
    Global State Redux: is user premium? userSlice.js
    i18n/Copy "What's on your mind?" label i18n/messages.js

    How Big Tech Teams Structure This

    A simplified directory structure inspired by Google or Meta:

    /components
  /PostForm
    PostForm.jsx                UI logic
    usePostForm.js              business logic + side effects
    PostValidation.js           validation logic
    PostUtils.js                utility logic
/services
  postApi.js                    RTK Query / API logic
/store
  userSlice.js                  global user state
/auth
  authUtils.js                  permission logic
/i18n
  postForm.en.json              labels & copy

    Why This Matters

    • Debug Faster You know where to look: bug in API? In formatting? In validation?
    • Team Collaboration Frontend, backend, design, and QA can work in parallel
    • Test Logic Separately You can write unit tests for each part
    • Swap Implementations Change API, UI library, or state logic without full rewrites
    • Consistency Every engineer follows the same architecture

    TL;DR

    You Already Knew... But Also Consider...
    UI vs Business Logic API logic (RTK, React Query)
    Auth, validation, effects
    Utilities, i18n, state machines
    Feature toggles, workflows

    Yes, UI logic and business logic are very different, especially in real-world large-scale codebases like Google, Facebook (Meta), Amazon, or any serious enterprise React application.

    Difference: UI Logic vs. Business Logic

    Concept UI Logic Business Logic
    Definition Controls how things look & behave visually Controls how data flows, rules, and app behavior
    Example Show a spinner while loading Fetch a user, calculate permissions, validate state
    Lives In React components Custom hooks, service layers, Redux slice, etc.
    Changes Often on UI tweaks (e.g. redesign) Often on product logic changes (e.g. pricing rules)
    Who Touches It Frontend developers, designers Frontend/backend/full-stack engineers
