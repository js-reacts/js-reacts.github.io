    Usage

    import React, { useEffect, useState, useRef } from "react";
import { Modal, Input, Button, message } from "antd";
import { useIdleTimer } from "react-idle-timer";

const IDLE_TIMEOUT = 1000 * 60; // 1 minute
const MODAL_TIMEOUT = 1000 * 30; // 30 seconds

function App() {
  const [isModalVisible, setIsModalVisible] = useState(false);
  const [password, setPassword] = useState("");
  const [attempts, setAttempts] = useState(0);
  const [countdown, setCountdown] = useState(30);
  const countdownRef = useRef(null);

  const forceLogout = () => {
    localStorage.removeItem("idleModalActive");
    message.error("You have been logged out.");
    window.location.href = "/login"; // or call logout logic
  };

  const handleAuth = async () => {
    // Replace with your real API call
    const success = password === "correct_password";

    if (success) {
      message.success("Re-authenticated!");
      setIsModalVisible(false);
      localStorage.removeItem("idleModalActive");
      resetModalState();
    } else {
      const newAttempts = attempts + 1;
      setAttempts(newAttempts);
      message.error("Incorrect password");

      if (newAttempts >= 2) {
        forceLogout();
      }
    }
  };

  const resetModalState = () => {
    setPassword("");
    setCountdown(30);
    setAttempts(0);
    clearInterval(countdownRef.current);
  };

  const showModal = () => {
    setIsModalVisible(true);
    localStorage.setItem("idleModalActive", "true");

    countdownRef.current = setInterval(() => {
      setCountdown((prev) => {
        if (prev <= 1) {
          clearInterval(countdownRef.current);
          forceLogout();
          return 0;
        }
        return prev - 1;
      });
    }, 1000);
  };

  const handleIdle = () => {
    const alreadyShown = localStorage.getItem("idleModalActive") === "true";
    if (!alreadyShown) {
      showModal();
    }
  };

  useIdleTimer({
    timeout: IDLE_TIMEOUT,
    onIdle: handleIdle,
    debounce: 500,
  });

  // Modal should persist after page refresh
  useEffect(() => {
    if (localStorage.getItem("idleModalActive") === "true") {
      showModal();
    }
    return () => clearInterval(countdownRef.current);
  }, []);

  return (
    <div style={{ padding: 20 }}>
      <h1>React Idle Session with Re-auth Modal</h1>

      <Modal
        title="Session Timeout"
        visible={isModalVisible}
        closable={false}
        footer={[
          <Button key="submit" type="primary" onClick={handleAuth}>
            Submit
          </Button>,
        ]}
      >
        <p>You’ve been inactive for 1 minute.</p>
        <p>
          Please enter your password to continue. You have {countdown}s left.
        </p>
        <Input.Password
          value={password}
          onChange={(e) => setPassword(e.target.value)}
          placeholder="Enter password"
        />
        <p style={{ color: "red", marginTop: 10 }}>
          {attempts > 0 && `Wrong attempt ${attempts}/2`}
        </p>
      </Modal>
    </div>
  );
}

export default App;

    ✅ What This IdleModal Does:

    • Listens to Redux state isIdleModalVisible
    • Shows an Ant Design modal when the user is idle
    • User must enter their password
    • Verifies password via API (or mock check)
    • Auto logs out after 30 seconds
    • Logs out after 2 failed attempts
    • Fully connected to your Redux state (authSlice)
    • Persisted across refresh with redux-persist

    🧪 Testing Tips

    • Try typing the wrong password twice → logs you out
    • Wait 30 seconds without entering anything → logs you out
    • Enter correct_password → modal closes and session continues

    isIdleModalVisible: false,

});

    Important

    Yes, you can combine <IdleSessionManager /> and <IdleModal /> into one single component — functionally it's fine.

    However, splitting them is generally better for maintainability.

    🔍 Here's the Reason Why They're Usually Separate

    Concern Explanation
    Separation of concerns <IdleSessionManager /> handles idle detection logic (timers, events).

    <IdleModal /> handles UI and re-authentication interaction. Keeping these separate makes both easier to test and maintain.
    Reusability You might want to reuse the modal logic elsewhere, or swap out the idle logic. Keeping them modular helps.
    Cleaner code Smaller components = less logic jammed into one file = easier debugging.
    Lifecycle clarity If you keep idle logic separate, you’re guaranteed that timer starts only once, and modal doesn’t accidentally remount.

// To:
dispatch(logoutFromIdle());

    Both Slice

    If you want both slices (idleSlice and authSlice) to coordinate during logout, here's how to do it cleanly:

    ✅ Goal

    When a user logs out due to idle timeout:

    • idleSlice resets the modal and attempts
    • authSlice clears tokens and user data
    • You dispatch both actions together from one place

    ✅ Option 1: Dispatch both actions from your component

    In your IdleModal (or combined idle manager), instead of:

    dispatch(logoutFromIdle());
dispatch(logoutUser());

    You can group them:

    dispatch(logoutFromIdle()); // reset idle state
dispatch(logoutUser()); // clear auth/session
window.location.href = "/login"; // optional redirect

    ✅ Option 2: Create a central logoutAll() action (if you want to abstract it)

    If you want to centralize the logic, create a custom thunk:

    🔧 In sessionActions.js or logoutHelpers.js

    import { logoutFromIdle } from "./idleSlice";
import { logoutUser } from "./authSlice";

export const logoutDueToIdle = () => (dispatch) => {
  dispatch(logoutFromIdle());
  dispatch(logoutUser());
  window.location.href = "/login"; // redirect if needed
};

    Then in your component:

    import { logoutDueToIdle } from "../redux/sessionActions";
dispatch(logoutDueToIdle());

    ✅ Summary

    Action Source Responsibility
    idleSlice Reset idle modal visibility and attempts
    authSlice Clear tokens/user session on logout
    Combined action Dispatch both together for full logout

    VVIP

    Is it appropriate to use api.dispatch(...) inside a component or request handler like this?

    Short Answer

    • Yes, if you're using RTK Query's baseQuery or custom async logic, then using api.dispatch(...) is correct and intended.

    • But if you're inside a React component, you should use useDispatch() from react-redux, not api.dispatch.

    🔍 Let's clarify with context:

    🧠 If you're inside a custom baseQuery, for example:

    const customBaseQuery = async (args, api, extraOptions) => {
  try {
    const result = await axios(...);
    return result;
  } catch (error) {
    if (error.status === 401) {
      const refreshResult = await axios.post(...);

      if (refreshResult.data) {
        api.dispatch(updateRefreshToken(refreshResult.data));
      } else {
        api.dispatch(logout());
        api.dispatch(resetState());
      }
    }
  }
}

    ✅ This is exactly where you should use api.dispatch — because RTK Query passes the api object (which includes dispatch) into your custom baseQuery.

    ⚠️ But if you're in a React component:

    Don't use api.dispatch(...).

    Use:

    import { useDispatch } from "react-redux";
const dispatch = useDispatch();

dispatch(logout());
dispatch(resetState());

    ✅ Fix for Your Example

    If your code is inside a baseQuery or mutation's onQueryStarted handler, this is fine:

    api.dispatch(updateRefreshToken(data));

    But if this is running in a React component, it should be:

    dispatch(updateRefreshToken(data)); // from useDispatch()

    🧠 Bonus Tip

    If you're refreshing tokens and retrying the original request, make sure you're doing it inside the baseQuery layer so it's seamless for every endpoint.

    Context Use api.dispatch(...)?
    RTK Query baseQuery ✅ Yes
    React component ❌ No → use useDispatch()
