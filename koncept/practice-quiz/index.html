    Concepts Quiz

    Here's a practice quiz to test your understanding of referential equality, shallow comparison, memoization, and related React concepts.

    Try answering each question, and I’ll provide explanations afterward.

    🧪 React Optimization Concepts Quiz

    1. Referential Equality

    What does this log?

    const a = { value: 1 };
const b = { value: 1 };

console.log(a === b);
    • A) true
    • B) false
    • C) Error
    answer

    Object reference comparison in JavaScript

    ✅ Correct Answer: B

    false

    ✅ Why:

    • In JavaScript, objects are compared by reference, not by value.
    • a and b may have the same content, but they are two different objects in memory.
    • So a === b returns false, because their references are not the same.
    const a = { value: 1 };
const b = { value: 1 };

console.log(a === b); // ❌ false

    📌 Takeaway:

    Even if two objects look identical, they are not equal unless they are the same reference.

    const a = { value: 1 };
const b = a;

console.log(a === b); // ✅ true — same reference

    2. Shallow Comparison

    Which of the following statements about shallow comparison is true?

    • A) It compares all nested properties deeply.
    • B) It uses === on top-level properties.
    • C) It compares values by type, not reference.
    Answer

    Understanding Shallow Comparison

    ✅ Correct Answer: B

    It uses === on top-level properties.

    ✅ Why:

    • Shallow comparison checks only the first level of properties using ===.
    • It does not recursively compare nested objects (that's deep comparison).

    • So for two objects, it will return true only if:

    • They have the same keys

    • Each top-level value is strictly equal (===)

    ❌ A is false

    • That would describe deep comparison, not shallow.

    ❌ C is false

    • === compares by reference for objects, not just by type or value.

    📌 Takeaway:

    Shallow comparison is fast but limited — great for simple prop changes, but not for detecting nested updates.

    3. React Re-Renders

    In this React component, how many times is handleClick created?

    function App() {
  const handleClick = () => console.log("clicked");
  return <button onClick={handleClick}>Click</button>;
}
    • A) Once, when the component is first rendered
    • B) Every time the component re-renders
    • C) Never — React reuses it
    Answer

    Function creation inside a component

    ✅ Correct Answer: B

    Every time the component re-renders

    ✅ Why:

    • Functions defined inside a component body (like handleClick) are re-created on every render.
    • Even though the function looks the same, it's a new reference each time.
    • This can lead to unnecessary re-renders in child components if the function is passed as a prop.

    🔥 How to optimize (if needed):

    Use useCallback to memoize the function:

    const handleClick = useCallback(() => {
console.log("clicked");
}, []);

    This keeps the same function reference between renders, unless dependencies change.

    📌 Takeaway:

    Functions declared inside components are not reused by default. They're re-created on every render unless wrapped with useCallback.

    4. useCallback Usage

    What does useCallback do?

    • A) Forces a function to run on every render
    • B) Memoizes a function so it keeps the same reference across renders
    • C) Automatically optimizes all React components
    Answer

    useCallback in React

    ✅ Correct Answer: B

    Memoizes a function so it keeps the same reference across renders

    ✅ Why:

    • useCallback returns a memoized version of the function that only changes if its dependencies change.
    • This helps prevent unnecessary re-renders, especially when passing functions to child components or using them in useEffect.

    ❌ A is false

    • useCallback does the opposite — it avoids creating a new function on every render.

    ❌ C is false

    • It only memoizes specific functions — it doesn’t globally optimize components.
    Example
    const handleClick = useCallback(() => {
console.log("clicked");
}, []);
    • This function will be reused across renders as long as the dependency array stays unchanged

    📌 Takeaway:

    useCallback is a performance optimization tool — it ensures stable function references across renders, avoiding unnecessary updates in child components.

    5. React.memo and Props

    Why might this component still re-render?

    const Button = React.memo(({ onClick }) => (
  <button onClick={onClick}>Click</button>
));

function App() {
  const handleClick = () => console.log("click");
  return <Button onClick={handleClick} />;
}
    • A) React.memo doesn't work with props
    • B) handleClick is re-created on each render
    • C) Button is not rendered at all
    Answer

    React.memo and Props

    ✅ Correct Answer: B

    handleClick is re-created on each render

    ✅ Why:

    • In React, functions declared inside a component (like handleClick in App) are re-created on every render.
    • Even though Button is wrapped in React.memo, it still re-renders because the onClick prop is technically different every time, even if it looks the same.
    • React.memo does a shallow comparison of props — and since the onClick function is a new reference, the component re-renders.

    ❌ A is false

    • React.memo does work with props, but it only skips re-rendering if props are shallowly equal.

    ❌ C is false

    • The Button is definitely rendered — it's just re-rendering unnecessarily due to a changing function prop.
    Example
    const Button = React.memo(({ onClick }) => (
    <button onClick={onClick}>Click</button>
));

function App() {
    const handleClick = useCallback(() => {
        console.log("click");
    }, []);

    return <Button onClick={handleClick} />;
}
    • ✅ Now, handleClick has a stable reference due to useCallback, so Button won’t re-render unnecessarily.

    📌 Takeaway:

    React.memo only prevents re-renders if props stay the same — use useCallback to memoize function props and avoid unnecessary updates.

    6. Immutable State

    Which state update is correct in React?

    const [user, setUser] = useState({ name: "Alice" });
    • A) user.name = "Bob"
    • B) setUser(user)
    • C) setUser({ ...user, name: "Bob" })
    Answer

    ✅ Correct Answer: C

    setUser({ ...user, name: "Bob" })

    ✅ Why:

    • In React, state should be treated as immutable.
    • Option C creates a new object with all existing user properties, but updates the name field — this triggers a re-render properly.

    ❌ A is false

    • user.name = "Bob" mutates the state directly, which is not allowed in React.
    • React won’t detect this change, so the component may not re-render.

    ❌ B is false

    • setUser(user) sets the same object reference, which React sees as "no change", so it skips re-rendering
    Example
    const [user, setUser] = useState({ name: "Alice" });

const updateName = () => {
    setUser({ ...user, name: "Bob" });
};
    • ✅ This ensures you're updating state immutably with a new object.

    📌 Takeaway:

    Always update React state immutably — create a new object or array rather than mutating the existing one.

    🧠 Advanced React Quiz (Expert Level)

    Here’s a more advanced React quiz focused on deeper topics like closure traps, stale state, memoization pitfalls, dependency management, and reconciliation. These are real-world challenges you'll face when building and optimizing complex React apps.

    1. useCallback & Changing Dependencies

    function Parent({ multiplier }) {
  const compute = useCallback(() => {
    return 5 * multiplier;
  }, []);

  return <Child compute={compute} />;
}

    What’s wrong with this code?

    • A) Nothing — useCallback memoizes the function.
    • B) The dependency array is empty, so multiplier won’t update.
    • C) compute will cause an infinite loop.
    • D) Child won’t receive the updated value of multiplier.
    Answer

    ✅ Correct Answer: B

    The dependency array is empty, so multiplier won’t update.

    ✅ Why:

    • useCallback(() => ..., []) tells React: "Never recreate this function unless something in the array changes."
    • But multiplier comes from props. If it changes, compute still uses the stale value (closure trap).

    🔥 Fix:

    const compute = useCallback(() => {
return 5 * multiplier;
}, [multiplier]);

    📌 Takeaway:

    Always include everything used inside the callback in the dependency array.

    2. useEffect with Non-Stable Dependencies

    useEffect(() => {
  console.log("Effect ran");
}, [{ a: 1 }]);

    What will happen?

    • A) The effect runs once.
    • B) The effect runs on every render.
    • C) You get a warning.
    • D) It depends on React’s version.
    Answer

    useEffect with Object Literal in Dependencies

    useEffect(() => {
console.log("Effect ran");
}, [{ a: 1 }]);

    ✅ Correct Answer: B

    The effect runs on every render.

    ✅ Why:

    • [ { a: 1 } ] is a new object every render.
    • Even though the contents look the same, React does a shallow comparison, and the reference is always different.

    🔥 Fix:

    Use useMemo or declare the object outside the render:

    const obj = useMemo(() => ({ a: 1 }), []);
useEffect(() => {
console.log("Effect ran");
}, [obj]);

    📌 Takeaway:

    Avoid inline objects/arrays in dependency arrays.

    3. Closure Trap

    function Counter() {
  const [count, setCount] = useState(0);

  const handleAlert = () => {
    setTimeout(() => {
      alert(count);
    }, 3000);
  };

  return <button onClick={handleAlert}>Alert Count</button>;
}

    If you click the button and then increment the count, what value does the alert show?

    • A) The latest count
    • B) Always 0
    • C) The count at the time you clicked
    • D) Depends on when setTimeout finishes
    Answer

    Closure Trap in setTimeout

    const handleAlert = () => {
    setTimeout(() => {
        alert(count);
    }, 3000);
};

    ✅ Correct Answer: C

    The count at the time you clicked.

    ✅ Why:

    • The setTimeout callback closes over the value of count when handleAlert was created.
    • Even if the count changes before the alert shows, the callback still "remembers" the old value.

    🔥 Fix:

    Use a ref to access the latest value:

    const countRef = useRef();
countRef.current = count;

setTimeout(() => {
alert(countRef.current);
}, 3000);

    📌 Takeaway:

    Closures capture values at the time they are defined, not live updates.

    4. useMemo Trap

    const memoizedValue = useMemo(() => heavyComputation(a, b), []);

    What issue might this cause?

    • A) heavyComputation runs on every render
    • B) a and b won't update memoizedValue
    • C) It causes memory leaks
    • D) It optimizes automatically
    Answer

    useMemo with Missing Dependencies

    const memoizedValue = useMemo(() => heavyComputation(a, b), []);

    ✅ Correct Answer: B

    a and b won't update memoizedValue.

    ✅ Why:

    • You passed an empty [] dependency array.
    • That means heavyComputation(a, b) only runs once — even if a or b change later.

    🔥 Fix:

    const memoizedValue = useMemo(() => heavyComputation(a, b), [a, b]);

    📌 Takeaway:

    useMemo won't re-run unless its dependencies change.

    Omitting needed dependencies causes stale or incorrect values — a classic optimization trap!

    5. React Reconciliation and Keys

    Why are keys important in a list?

    {
  items.map((item) => <div key={item.id}>{item.name}</div>);
}
    • A) To style list items
    • B) To identify which DOM elements changed
    • C) So React can rerender faster
    • D) Both B and C
    Answer 
    items.map((item) => <div key={item.id}>{item.name}</div>);

    ✅ Correct Answer: D

    Both B) Identify DOM changes and C) Improve re-render performance

    ✅ Why:

    • Keys give React a stable identity for each element in the list.

    • During reconciliation, React uses these keys to:

      • Detect what’s changed, added, or removed between renders (B)
      • Avoid re-rendering or re-creating elements unnecessarily, improving performance (C)

    🔥 Common Mistake:

    // ❌ Can cause bugs in dynamic lists
items.map((item, index) => <div key={index}>{item.name}</div>);

    ✅ Use unique, stable identifiers (like item.id)

    📌 Takeaway:

    Keys are critical for efficient diffing and correct rendering.

    They help React do less work and avoid UI bugs when lists change.

    6. React.memo with props mutation

    What happens here?

    const data = { value: 42 };

const Child = React.memo(({ data }) => {
  useEffect(() => {
    console.log("Effect ran");
  }, [data]);

  return <div>{data.value}</div>;
});

<Child data={data} />;
    • A) Effect only runs once
    • B) Effect runs on every render
    • C) Effect won’t run because memo blocks it
    • D) Depends on React.memo config
    Answer

    React.memo doesn’t prevent prop reference changes

    ✅ Correct Answer: B

    Effect runs on every render

    ✅ Why:

    • Even though React.memo memoizes the component, it performs a shallow comparison of props.
    • On every render, a new object ({ value: 42 }) is created for data, meaning data !== previousData.
    • So React.memo thinks props have changed, and re-renders the component.
    • As a result, the useEffect also re-runs because data (by reference) has changed.

    🔥 Fix:

    Avoid creating new objects inline unless necessary:

    const memoizedData = useMemo(() => ({ value: 42 }), []);
<Child data={memoizedData} />;

    📌 Takeaway:

    Object identity matters.

    React.memo and useEffect rely on reference equality — not deep value comparison.

    New object ===new reference=== re-render and re-effect.

    import React, { useEffect, useMemo } from "react";

const Child = React.memo(({ data }) => {
    useEffect(() => {
        console.log("Effect ran");
    }, [data]);

    return <div>{data.value}</div>;
});

function App() {
    // useMemo ensures `data` object has the same reference unless its contents change
    const data = useMemo(() => ({ value: 42 }), []);

    return (
        <div>
            <h1>Parent Component</h1>
            <Child data={data} />
        </div>
    );
}

export default App;
